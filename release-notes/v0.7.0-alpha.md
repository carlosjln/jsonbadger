JsonBadger 0.7.0-alpha reworks PostgreSQL capability binding and identity policy around one bound schema runtime. This release replaces id_strategy with identity, restores database-generated numeric ids as the default path, and keeps PostgreSQL feature selection behind internal runtime binding instead of per-query branching. These set of changes enable the support for PostgreSQL 9.6 implementation of json fields.

## Example

```js
import JsonBadger from 'jsonbadger';
import {v7 as uuidv7} from 'uuid';

const UserSchema = new JsonBadger.Schema({
	name: String
});

const SessionSchema = new JsonBadger.Schema({
	token: String
}, {
	identity: {
		type: JsonBadger.IDENTITY_TYPE.uuid,
		format: JsonBadger.IDENTITY_FORMAT.uuidv7,
		mode: JsonBadger.IDENTITY_MODE.fallback,
		generator: uuidv7
	}
});

await User.find({
	'payload.profile.city': {$exists: true}
}).exec();
```

> **Note:** `id_strategy` is no longer the active schema API. Configure UUID-backed ids through `identity`.

## What's Included

1. Capability-bound schema runtime: `connect(...)` now scans factual PostgreSQL capabilities once, compiled models bind them once through `schema.$bind_connection(connection)`, and downstream code consumes `model.schema.$runtime`.
2. Identity policy replacement: public identity configuration now uses `identity.type`, `identity.format`, `identity.mode`, and `identity.generator`, with `IDENTITY_TYPE`, `IDENTITY_FORMAT`, and `IDENTITY_MODE` exported for callers.
3. Restored numeric default: the library default is back to database-generated `bigint` ids with phase-one `BIGSERIAL PRIMARY KEY` DDL.
4. UUIDv7 runtime modes: UUID-backed ids now support `fallback`, `database`, and `application` binding, with native database generation only when the connected server can actually support it.
5. Bound DDL and insert behavior: table creation, insert generation, explicit-id rejection, and id casting now read from `schema.$runtime.identity` instead of a global strategy flag.
6. Stable public id shape: `document.id` remains string-shaped across numeric and UUID-backed modes.
7. Clean path-existence surface: `$exists` is the documented path-existence operator, with native JSONPath used internally on newer PostgreSQL versions and SQL compatibility used on older servers.
8. Synced docs and tests: examples, API docs, query translation notes, and focused unit and integration coverage were updated to match the current runtime.

## Release Notes

1. `Schema` now owns capability-finalized runtime artifacts under `schema.$runtime`, including bound read operators and bound identity behavior.
2. `Model` compilation now clones the source schema, binds it to the current `Connection`, and uses that finalized clone as the runtime truth for query, insert, and migration behavior.
3. `Connection` capability detection is factual only. It reports values such as `server_version`, `server_version_num`, `supports_uuidv7`, and `supports_jsonpath`, but it does not choose product behavior on its own.
4. Public identity behavior now separates storage type, format, and generation mode instead of folding them into one `id_strategy` value.
5. Supported phase-one identity combinations are:
   A. `bigint + null format`
   B. `uuid + uuidv7 + fallback`
   C. `uuid + uuidv7 + database`
   D. `uuid + uuidv7 + application`
6. `$exists` is the normal public path-existence operator. PostgreSQL `12+` may bind a native JSONPath-backed implementation internally, while PostgreSQL `9.6` uses a SQL compatibility fallback for the same public query.
7. `defaults` is now part of the public package surface. Consumers can set repo-wide schema defaults such as `JsonBadger.defaults.schema_options.identity` before constructing schemas.

## Upgrade Notes

1. Replace schema-level `id_strategy` usage with `identity`.
2. If your application relied on implicit UUIDv7 defaults, declare UUID identity explicitly with `identity.type`, `identity.format`, `identity.mode`, and `identity.generator`.
3. Expect the default schema identity to be numeric and database-generated unless you opt into UUID identity explicitly.
4. Use `$exists` as the public path-existence operator. Do not depend on JSONPath-shaped public operators.
